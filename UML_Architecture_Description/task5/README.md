Мета системи — автоматизувати процеси роботи бібліотеки: пошук та перегляд книг, бронювання, видача та повернення, а також управління каталогом адміністратором. 
Система повинна бути зрозумілою для користувачів і легко підтримуваною для розробників.

Тип архітектури
Обрана архітектура — Layered (багатошарова). Вона розділяє систему на три основні рівні:

Presentation (UI) — інтерфейс для користувачів та адміністраторів.

Business Logic — сервіси та правила роботи бібліотеки.

Data Access — робота з базою даних та сховищем інформації.

Основні компоненти
UI (LibraryUI, AdminUI) — відображає дані та приймає запити від користувачів.

Business Logic (UserService, BookService, ReservationService) — реалізує правила: бронювання, видача, повернення книг.

Data Access (UserRepository, BookRepository, ReservationRepository) — відповідає за збереження та отримання даних.

Використані UML-діаграми
Component Diagram — показує залежності між шарами (UI → Services → Repositories).

Use Case Diagram — демонструє основних акторів (Користувач, Адміністратор) та їх сценарії (взяти книгу, повернути книгу, додати книгу).

Sequence Diagram — деталізує сценарій «Взяти книгу» (користувач → UI → сервіси → репозиторії).

State Machine Diagram — описує життєвий цикл книги (Доступна → Заброньована → Видана → Повернена).

Коротке пояснення
UI шар відповідає лише за взаємодію з користувачем, не містить бізнес‑логіки.

Business Logic шар реалізує правила бібліотеки, координує роботу між UI та даними.

Data Access шар ізолює роботу з базою даних, що дозволяє легко змінювати технологію збереження.

Такий підхід забезпечує зрозумілу структуру, простоту підтримки та розширення системи.
